## 第1轮：信息架构评审

### 交易团队意见

**用户打开App后最需要看到的3个信息：**

1. **今天投不投、投多少份** -- 这是定投用户的核心决策，应该在第一屏顶部直接给出结论
2. **当前PE档位和纳指价格** -- 用户需要知道"为什么"投这么多，PE档位是核心依据
3. **距下次定投日还有几天** -- 用户需要知道时间窗口，决定是否需要现在操作

**当前信息层级问题：**

- `app.js` 第1009-1144行 `renderTodayTab()` 中，信息排列顺序为：心理锚点 -> 环境快照 -> 操作指令 -> 事件日历 -> 趋势判断 -> 数据说服
- **核心问题**：操作指令（用户最关心的"投多少份"）被放在了第三张卡片（第1074行），前面有大段的心理锚点文字和环境快照数据。用户需要滚动约2屏才能看到决策结论
- 心理锚点卡片（第1016-1026行）包含大量文字，信息密度过高，作为首屏内容不够精炼
- 环境快照6宫格（第1031-1062行）展示了6个数据项，但用户打开App时并不需要同时看到VIX、美债收益率、20日均线等全部数据

**代码行引用：**
- `app.js:1009-1013` -- alert banner 位置合理，保留
- `app.js:1016-1026` -- 心理锚点卡片，建议移至操作指令之后
- `app.js:1074-1105` -- 操作指令卡片，应提升至首屏第一张卡片

### Web团队意见

**卡片排列顺序问题：**

- 当前首屏视觉层级：心理锚点（大段文字）-> 环境快照（6格数据）-> 操作指令（带边框高亮卡片）
- 操作指令卡片虽然有 `action-card` 样式和彩色边框（`index.html:273-321`），但因为位置靠后，用户第一眼看不到
- 心理锚点卡片使用了 `quote-card` 的渐变背景（`index.html:414-419`），视觉权重很高，但内容并非最高优先级

**信息密度问题：**

- 环境快照的6宫格（`index.html:189-193`，`data-grid` 2列布局）在小屏手机上信息密度偏高
- 6个数据项中，"距周四定投"和"20日均线"对当日决策的重要性较低，不应与PE、价格并列
- 心理锚点卡片内文字约150字（`app.js:1018-1025`），在手机上需要滚动阅读

**视觉层级：**

- Tab导航栏（`index.html:762-768`）有5个Tab，在小屏上可能需要横向滚动，"今日决策"作为默认Tab是正确的
- Header区域（`index.html:746-759`）信息合理：标题 + 状态徽章 + 刷新/设置按钮

### 共同决策

**推荐的信息展示顺序（首屏）：**

1. **Alert Banner**（非交易日/周四/极端行情提示）-- 保留现有位置
2. **操作指令卡片**（今天投多少份）-- 从第3张提升到第1张
3. **PE仪表盘 + 核心数据**（PE档位 + 纳指价格，精简为3项关键数据）
4. **心理锚点**（移至操作指令之后，作为"为什么这样操作"的解释）
5. **趋势判断 + 事件日历**（折叠或移至第二屏）
6. **数据说服**（移至底部，作为收尾激励）

**必须执行的改进：**

| # | 改进项 | 代码位置 | 优先级 |
|---|--------|----------|--------|
| 1 | 将操作指令卡片移至首屏第一张卡片 | `app.js:1009-1144` renderTodayTab() | P0 |
| 2 | 精简心理锚点文字，从约150字压缩到50字以内 | `app.js:1016-1026` | P1 |

## 第2轮：决策效率评审

### 交易团队意见

**从打开App到做出决策需要几步？**

当前流程：
1. 打开App -> 看到心理锚点文字（需要阅读理解）
2. 滚动 -> 看到环境快照6格数据（需要逐个阅读）
3. 继续滚动 -> 看到操作指令卡片（终于看到"投多少份"）
4. 在操作指令卡片中找到"今日操作"行（第1097-1104行）-> 看到总份数

**结论：需要3步滚动 + 多次视觉扫描，无法1步到位。**

**关键问题：**

- `app.js:1097-1104` 中"今日操作：基准1份"和最终份数的展示不够醒目。份数用了 `font-size:18px`，但被包裹在一个普通的 flex 布局中，没有视觉强调
- 操作指令卡片内部信息层级不清晰：周四定投份数 -> 额外加仓 -> 今日总操作，用户需要理解三段信息才能得到最终答案
- 加仓判定逻辑（`app.js:622-779`）的三步法（跌幅判定 -> 估值修正 -> 事件降档）在UI中全部以文字展示（`app.js:1093` action-reason），信息过载

**代码行引用：**
- `app.js:1078-1104` -- 操作指令卡片内部布局，需要重构为"结论先行"
- `app.js:1097-1104` -- "今日操作"行，应作为卡片最顶部、最大字号的元素
- `app.js:622-779` -- calculateAdditionalBuy() 三步法逻辑，UI展示应简化

### 共同决策

## 第4轮：角色系统UX评审

### 交易团队意见

**多角色切换场景是否合理：**

- 角色系统的设计意图是支持"家庭账户"或"多策略对比"场景（`app.js:35-46` DEFAULT_PROFILE）
- 对于大多数个人定投用户，只需要1个角色（默认账户）
- 可能需要多角色的场景：
  - 夫妻各自定投，想分开记录
  - 同时运行"激进策略"和"保守策略"进行对比
  - 为子女设立教育基金定投账户

**当前角色系统的问题：**

- `app.js:131-150` `addProfile()` -- 新角色默认参数与默认角色完全相同（startDate=今天, frequency=weekly, baseShares=1, assetRatio=1.0），用户创建新角色后需要手动修改所有参数，但没有引导
- `app.js:184-206` `saveCurrentProfile()` -- 保存后使用 `alert('角色已保存')`（第205行），体验粗糙
- `app.js:99-112` `ensureDefaultHistoryRecord()` -- 默认角色硬编码了一条2026-06-05的定投记录（`app.js:48-56`），这个设计意味着所有用户打开App就有一条历史记录，可能造成困惑
- 角色切换后没有视觉反馈告知用户"当前查看的是哪个角色的数据"

**代码行引用：**
- `app.js:35-46` -- DEFAULT_PROFILE 定义
- `app.js:48-56` -- DEFAULT_HISTORY_RECORD 硬编码
- `app.js:131-150` -- addProfile()
- `app.js:172-182` -- switchProfile()
- `app.js:184-206` -- saveCurrentProfile()

### Web团队意见

**角色管理入口位置：**

- 角色管理藏在设置面板中（`index.html:819-864`），需要点击齿轮按钮 -> 设置面板 -> 角色管理区域，入口深度为3级
- 对于只有1个角色的用户，这个入口几乎不会被使用，占用了设置面板的顶部空间
- 对于有多角色需求的用户，每次切换角色都需要：打开设置 -> 选择角色 -> 关闭设置，操作路径过长

**交互流程问题：**

- `index.html:824` -- 角色选择使用 `<select>` 下拉框，在移动端体验不佳（下拉框在小屏上触发原生选择器，样式不可控）
- `index.html:858-859` -- "保存角色"和"删除角色"按钮紧挨着，容易误触
- 角色编辑区域（`index.html:828-860`）始终显示，即使只有一个不可编辑的默认角色，也显示了所有编辑字段
- 缺少角色切换的快捷方式：Header中没有显示当前角色名称，用户不知道自己在查看哪个角色的数据

**设置面板结构问题：**

- `index.html:866-873` -- "全局设置"区域中的"定投占比系数"与角色编辑区域中的"定投占比系数"（`index.html:854-856`）功能重复，容易混淆
- `app.js:1631-1649` `loadSettings()` 和 `saveSettings()` -- 全局设置和角色设置分别存储在 `nasdaq_dca_settings` 和 `nasdaq_dca_profiles` 两个 localStorage key 中，但全局设置的 `totalAssets` 和角色的 `assetRatio` 功能重叠

### 共同决策

**角色系统优化建议：**

1. **Header中显示当前角色名称**：在标题旁或下方显示当前角色名，点击可快速切换
2. **简化单角色体验**：当只有默认角色时，隐藏角色管理区域，只显示全局设置
3. **角色切换改为顶部下拉菜单或底部Tab**：减少操作路径
4. **消除全局设置与角色设置的重复**：将"定投占比系数"统一到角色设置中，移除全局设置中的重复项
5. **新角色创建引导**：创建新角色时弹出引导对话框，帮助用户设置差异化参数

**必须执行的改进：**

| # | 改进项 | 代码位置 | 优先级 |
|---|--------|----------|--------|
| 1 | Header中显示当前角色名称，点击可快速切换 | `index.html:746-759`, `app.js` 新增 | P1 |

## 第5轮：整体体验评审

### 交易团队意见

**策略完整性评估：**

**优点：**
- PE档位体系（`app.js:612-619`）设计合理，6档覆盖了从低估到高估的完整区间
- 加仓三步法（`app.js:622-779`）逻辑严谨：跌幅判定 -> 估值修正 -> 事件降档，层层递进
- 周度加仓上限（`app.js:723-747`）防止过度加仓，是重要的风控机制
- 连续加仓检测（`app.js:750-769`）避免情绪化连续操作
- 事件降档机制（`app.js:670-703`）考虑了FOMC、CPI、非农等关键事件
- 黑天鹅保护（`app.js:694-697`）：单日跌>7%跳过加仓，防止抄底失败

**不足：**
- `app.js:563-599` `estimatePE()` -- PE估算基于线性插值，精度有限。已知数据点只到2025-06（22000点/34.5倍PE），对于2026年的价格外推可能不准确
- `app.js:602-609` `generateHistoricalPE()` -- 历史PE生成使用了 `Math.random()`（第607行），每次刷新结果不同，不应用于趋势分析
- `app.js:1519-1529` `calculatePEPercentile()` -- PE百分位基于正态分布近似（mean=27, std=6），而非真实历史数据，精度存疑
